# 上下文退化速查

> 一句话总结：上下文越大，模型表现越差，需要在合适时机干预。

---

## 一、五大问题

| 问题 | 现象 | 通俗理解 |
|-----|------|---------|
| **中间丢失** | 上下文中间的信息被忽略 | "说了但没听见" |
| **中毒** | 错误信息累积，越错越离谱 | "谣言传多了变真理" |
| **分心** | 无关信息干扰，模型注意力分散 | "信息太多看花眼" |
| **混淆** | 不同任务的信息混在一起 | "张冠李戴" |
| **冲突** | 多条信息互相矛盾 | "不知道听谁的" |

---

## 二、解决方案速查

### 1. 中间丢失

**问题**：关键信息放在中间，模型容易忽略。

**解决**：
- 关键信息放开头或结尾
- 用醒目的标记（如 `[重要]`）
- 减少中间区域的冗余内容

```
好的结构：
[当前任务] ← 开头，高注意力
- 目标：xxx
- 截止：xxx

[详细数据] ← 中间，低注意力（放不重要的）
- ...

[关键结论] ← 结尾，高注意力
- 结论：xxx
```

---

### 2. 中毒

**问题**：错误信息进入上下文，被反复引用，越错越离谱。

**解决**：
- 监控错误信息数量（超过阈值就清理）
- 定期验证关键信息
- 发现问题后截断到出错点之前

```python
# 简单的中毒预防
if count_errors(context) > 5:
    truncate_to_before_last_error()
```

---

### 3. 分心

**问题**：无关信息太多，模型注意力被分散。

**解决**：
- 只放相关信息（相关性过滤）
- 用命名空间区分不同类型
- 不确定是否需要的信息，不放

```
不好的做法：
上下文包含：用户问题 + 产品文档 + 竞品分析 + 历史对话 + ...

好的做法：
上下文只包含：用户问题 + 相关的产品文档片段
```

---

### 4. 混淆

**问题**：不同任务的信息混在一起，模型用错上下文。

**解决**：
- 不同任务用不同上下文窗口
- 任务切换时明确标记
- 用子 Agent 隔离不同任务

```
# 任务切换时明确标记
[任务A结束]

========== 切换到任务B ==========

[任务B开始]
```

---

### 5. 冲突

**问题**：多条信息互相矛盾，模型不知道用哪个。

**解决**：
- 标记信息来源和时效性
- 设置优先级规则
- 过滤过时信息

```
# 标记来源和时效
[2024-01-15 产品文档] 价格：99元
[2024-03-20 内部通知] 价格调整为：89元  ← 优先使用
```

## 三、代码检测方法（scripts 里怎么做的）

### 核心思路：启发式估计，不是精确检测

```
┌────────────────────────────────────────────────────┐
│  检测方法 = 简单规则 + 阈值判断                      │
│                                                    │
│  不需要精确，只需要"足够好的估计"                    │
└────────────────────────────────────────────────────┘
```

### 1. 中间丢失检测

```python
# 方法：看位置
def detect_lost_in_middle(position, total):
    # 前10%和后10%是"安全区"
    if position < total * 0.1 or position > total * 0.9:
        return "safe"
    else:
        return "at_risk"  # 中间80%有风险
```

**原理**：研究证明模型注意力呈 U 型，中间位置注意力低。

---

### 2. 中毒检测

```python
# 方法：数错误关键词
error_patterns = ["error", "failed", "exception", "cannot", "unable"]

def detect_poisoning(context):
    error_count = count_matches(context, error_patterns)
    
    if error_count > 5:
        return "高风险"
    elif error_count > 3:
        return "中风险"
    else:
        return "低风险"
```

**原理**：错误信息越多，上下文越可能被污染。

---

### 3. 退化检测

```python
# 方法：算利用率
def check_degradation(token_count, limit):
    utilization = token_count / limit
    
    if utilization > 0.9:
        return "危急"  # 必须处理
    elif utilization > 0.7:
        return "退化"  # 需要关注
    elif utilization > 0.5:
        return "警告"  # 开始注意
    else:
        return "健康"
```

**原理**：上下文越大，性能越差。

---

### 4. 分心检测

```python
# 方法：算相关性分数
def detect_distraction(context_elements, task):
    for element in context_elements:
        relevance = similarity(element, task)
        if relevance < 0.3:  # 低相关性
            mark_as_distractor(element)
```

**原理**：无关信息会争夺模型的注意力。

## 四、实用建议

### 预防 > 检测

| 策略 | 做法 | 效果 |
|-----|------|------|
| 控制大小 | 设置保守的 token 上限 | 避免退化 |
| 放对位置 | 关键信息放两端 | 避免丢失 |
| 定期清理 | 每 N 轮压缩一次 | 避免中毒 |
| 任务隔离 | 不同任务用子 Agent | 避免混淆 |

### 阈值建议

```python
# 推荐设置（根据实际情况调整）
WARNING = 0.5   # 50% 利用率，开始注意
COMPACT = 0.7   # 70% 利用率，执行压缩
CRITICAL = 0.85 # 85% 利用率，必须处理
```

### 什么时候干预？

```
上下文利用率 < 50%：不用管
上下文利用率 50-70%：开始监控
上下文利用率 70-85%：执行压缩
上下文利用率 > 85%：必须处理（压缩/隔离/重启）
```

---

## 五、一句话总结

> **上下文退化无法精确检测，但可以通过简单规则预防。核心原则：控制大小、放对位置、定期清理、任务隔离。**
