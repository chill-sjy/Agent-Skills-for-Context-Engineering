# 上下文压缩策略详解

## 概述

当智能体会话产生数百万 token 的对话历史时，上下文压缩成为必需。但压缩不仅仅是减少 token 数量——错误的压缩策略会导致智能体"遗忘"关键信息，从而需要重新获取数据，最终消耗更多 token。

**核心洞察**：优化目标不是"每次请求的 token 数"（tokens-per-request），而是"每次任务的 token 数"（tokens-per-task）——即完成整个任务从开始到结束所消耗的总 token 数，包括因信息丢失而产生的重新获取成本。

本文档详细解析三种生产级上下文压缩策略，基于 36,611 条消息的评估数据，揭示它们的工作原理、性能差异和适用场景。

---

## 三种压缩策略

### 1. 锚定迭代摘要 (Anchored Iterative Summarization)

#### 工作原理

锚定迭代摘要维护一个**结构化的持久摘要**，包含固定的章节（如会话意图、文件修改、决策、下一步等）。关键特点：

- **首次压缩**：当上下文达到阈值（通常 70-80% 利用率）时，将被截断的历史记录摘要化到预定义的章节中
- **后续压缩**：只摘要新截断的内容，然后**增量合并**到现有摘要中，而不是重新生成整个摘要
- **结构强制保留**：每个章节就像一个检查清单，摘要器必须填充，防止信息静默丢失

#### 示例说明

假设一个调试会话有 178 条消息（89,000 tokens）：

**原始对话片段**：
```
User: /api/auth/login 返回 401 错误
Assistant: 让我检查认证控制器...
[读取 auth.controller.ts]
Assistant: JWT 生成看起来正常，检查会话存储...
[读取 config/redis.ts]
Assistant: 发现问题！Redis 连接配置陈旧，导致会话无法持久化
[修改 config/redis.ts - 添加连接池]
[修改 services/session.service.ts - 添加重试逻辑]
Assistant: 运行测试... 14 个通过，2 个失败
```

**锚定迭代摘要输出**：
```markdown
## Session Intent
调试 /api/auth/login 端点的 401 未授权错误，尽管凭据有效

## Root Cause
会话存储中的 Redis 连接陈旧。JWT 生成正确但会话无法持久化

## Files Modified
- auth.controller.ts: 未修改（仅读取）
- middleware/cors.ts: 未修改（已检查）
- config/redis.ts: 修复连接池配置
- services/session.service.ts: 为瞬态故障添加重试逻辑
- tests/auth.test.ts: 更新模拟设置

## Test Status
14 个通过，2 个失败（模拟设置问题）

## Next Steps
1. 修复剩余测试失败（模拟会话服务）
2. 运行完整测试套件
3. 部署到预发布环境
```

**为什么有效**：
- "Files Modified" 章节强制列出所有文件操作，防止遗忘
- "Root Cause" 章节保留技术细节（Redis、JWT、会话持久化）
- "Test Status" 章节保留具体数字，而不是模糊的"大部分通过"

#### 增量合并机制

第二次压缩时，假设又有 50 条新消息：

```
User: 测试还是失败
Assistant: 检查模拟配置...
[修改 tests/auth.test.ts]
Assistant: 修复了模拟问题，现在 16 个全部通过
```

**合并过程**：
1. 只摘要这 50 条新消息
2. 将新信息合并到现有章节：
   - "Files Modified" 更新 `tests/auth.test.ts` 的描述
   - "Test Status" 更新为 "16 个全部通过"
   - 保留之前的 Root Cause 和其他历史信息

这避免了重新处理整个历史，减少了信息漂移。

### 2. 不透明压缩 (Opaque Compression)

#### 工作原理

不透明压缩生成**针对重建保真度优化的压缩表示**，而不是人类可读的摘要。类似于数据压缩算法（如 gzip），它追求最高压缩率，但牺牲可解释性。

**特点**：
- 压缩率最高（99.3%，相比锚定迭代的 98.6%）
- 输出不可读，无法验证保留了什么信息
- 依赖模型的内部表示能力
- 适合短会话或重新获取成本低的场景

#### 示例说明

同样的调试会话，不透明压缩可能产生：

```
[COMPRESSED_CONTEXT]
auth_login_401|redis_stale_conn|files:config/redis.ts+mod,services/session.service.ts+mod,auth.controller.ts+read|tests:14pass_2fail->16pass|next:deploy_staging
[/COMPRESSED_CONTEXT]
```

这是一个极简化的示例。实际的不透明压缩可能使用：
- Token ID 序列
- 嵌入向量
- 模型特定的压缩编码

**优势**：节省更多 token（额外 0.7%）

**劣势**：
- 无法人工检查是否保留了关键信息
- 如果压缩丢失了某个文件路径，你无法提前发现
- 依赖模型的解压能力，可能在长会话中累积误差

---

### 3. 再生完整摘要 (Regenerative Full Summary)

#### 工作原理

每次压缩触发时，**从头重新生成**整个摘要，而不是增量合并。

**特点**：
- 每次都审查完整的上下文历史
- 产生连贯、可读的摘要
- 但在多次压缩周期中可能丢失细节

#### 示例说明

**第一次压缩**（处理前 100 条消息）：
```markdown
## Summary
用户报告 /api/auth/login 的 401 错误。我们追踪到 Redis 连接问题，
修改了 config/redis.ts 和 services/session.service.ts。
测试状态：14 通过，2 失败。
```

**第二次压缩**（处理前 100 条消息 + 新的 50 条消息）：
```markdown
## Summary
用户报告认证错误。我们修复了 Redis 配置和会话服务。
所有测试现在通过。准备部署。
```

**问题**：注意第二次摘要丢失了：
- 具体的端点路径 `/api/auth/login`
- 具体的错误代码 `401`
- 中间的测试状态 `14 通过，2 失败`

**为什么会丢失**：
- 模型重新审查整个历史时，倾向于"总结"而非"保留细节"
- 早期的技术细节在后续摘要中被视为"已解决"而省略
- 没有结构强制保留特定类型的信息

## 评估数据对比

### 评估方法：基于探针的测试

传统的文本相似度指标（如 ROUGE、嵌入余弦相似度）无法衡量功能性压缩质量。一个摘要可能在词汇重叠上得分很高，但恰好遗漏了智能体需要的那个文件路径。

**探针测试**直接测量功能质量：压缩后向智能体提问，看它能否正确回答。

#### 四种探针类型

| 探针类型 | 测试内容 | 示例问题 |
|---------|---------|---------|
| **召回探针** (Recall) | 事实保留 | "原始错误消息是什么？" |
| **工件探针** (Artifact) | 文件追踪 | "我们修改了哪些文件？每个文件改了什么？" |
| **延续探针** (Continuation) | 任务规划 | "接下来应该做什么？" |
| **决策探针** (Decision) | 推理链 | "关于 Redis 问题我们做了什么决定？为什么？" |

**评分逻辑**：
- 如果压缩保留了正确信息，智能体能准确回答
- 如果信息丢失，智能体会猜测或产生幻觉
- 使用 LLM-as-judge 对回答质量打分（0-5 分）

### 六个评估维度

| 维度 | 测量内容 | 关键问题 |
|-----|---------|---------|
| **准确性** (Accuracy) | 技术细节正确性 | 文件路径、函数名、错误代码是否准确？ |
| **上下文感知** (Context Awareness) | 对话状态理解 | 响应是否反映当前会话状态？ |
| **工件追踪** (Artifact Trail) | 文件操作记忆 | 智能体是否知道读取/修改了哪些文件？ |
| **完整性** (Completeness) | 回答覆盖度 | 是否解决了问题的所有部分？ |
| **连续性** (Continuity) | 无缝继续能力 | 能否在不重新获取信息的情况下继续工作？ |
| **指令遵循** (Instruction Following) | 约束遵守 | 响应是否遵守声明的约束？ |

### 性能对比表

基于 36,611 条消息、数百个压缩点的评估结果：

| 方法 | 总分 | 准确性 | 上下文感知 | 工件追踪 | 完整性 | 连续性 | 指令遵循 |
|-----|------|--------|----------|---------|--------|--------|----------|
| **锚定迭代** | **3.70** | **4.04** | **4.01** | **2.45** | 4.44 | 3.80 | 4.99 |
| **再生完整** | 3.44 | 3.74 | 3.56 | 2.33 | 4.37 | 3.67 | 4.95 |
| **不透明** | 3.35 | 3.43 | 3.64 | 2.19 | 4.37 | 3.77 | 4.92 |

**压缩率对比**：

| 方法 | 压缩率 | 质量分数 | Token 节省 vs 质量权衡 |
|-----|--------|---------|---------------------|
| 锚定迭代 | 98.6% | 3.70 | 保留 1.4% token，换取最佳质量 |
| 再生完整 | 98.7% | 3.44 | 中等压缩，中等质量 |
| 不透明 | 99.3% | 3.35 | 最高压缩，质量损失最大 |

### 关键发现

1. **准确性差距最大**：锚定迭代比不透明压缩高 0.61 分（15% 提升）
   - 结构化章节强制保留技术细节
   - 不透明压缩在重建时可能丢失精确的文件路径或错误代码

2. **上下文感知差距**：锚定迭代比再生完整高 0.45 分
   - 增量合并保留历史决策的上下文
   - 完全再生倾向于"总结"而非"保留"

3. **工件追踪普遍较弱**：所有方法都在 2.19-2.45 范围（满分 5.0）
   - 这是所有压缩方法的共同弱点
   - 即使有专门的 "Files Modified" 章节，长会话中仍难以完整追踪
   - 建议：对于文件追踪至关重要的场景，考虑单独维护工件日志

4. **完整性和指令遵循差异小**：所有方法都在 4.37-4.44 和 4.92-4.99
   - 这些维度主要取决于基础模型能力，而非压缩策略

5. **Token 节省 vs 质量权衡**：
   - 不透明压缩额外节省 0.7% token（相比锚定迭代）
   - 但质量损失 0.35 分（10% 下降）
   - 对于任何重新获取成本重要的任务，这个权衡不值得

### 统计显著性

- 差异在不同任务类型（调试、功能实现、代码审查）中一致
- 差异在短会话和长会话中都保持稳定
- 样本量：36,611 条消息，数百个压缩点
- 差异范围 0.26-0.35 分具有统计显著性

## 适用场景分析

### 何时使用锚定迭代摘要

**最佳场景**：
- ✅ 长时间运行的会话（100+ 消息）
- ✅ 文件追踪至关重要（编码、调试任务）
- ✅ 需要验证保留了什么信息
- ✅ 重新获取成本高（大型代码库、复杂系统）
- ✅ 多次压缩周期（会话可能触发 3+ 次压缩）

**实际案例**：
- 调试复杂的分布式系统问题
- 大型代码库的重构任务
- 需要追踪多个文件修改的功能开发
- 长期的架构设计讨论

**实现要点**：
```markdown
## 定义结构化章节
- Session Intent: 用户目标
- Root Cause: 问题根源（调试场景）
- Files Modified: 文件操作详细列表
- Decisions Made: 关键决策和理由
- Current State: 当前进度
- Next Steps: 待办事项

## 压缩流程
1. 首次压缩：摘要截断历史到各章节
2. 后续压缩：只摘要新内容，增量合并
3. 追踪信息来源（哪个压缩周期）以便调试
```

---

### 何时使用不透明压缩

**最佳场景**：
- ✅ 需要最大 token 节省
- ✅ 会话相对较短（< 50 消息）
- ✅ 重新获取成本低（小型项目、简单任务）
- ✅ 单次或少量压缩周期
- ✅ 不需要人工检查压缩内容

**实际案例**：
- 简单的代码生成任务
- 单文件修改
- 快速原型开发
- 成本敏感的应用（需要最小化 API 调用成本）

**权衡考虑**：
- 额外节省 0.7% token
- 但质量下降 10%
- 如果任务失败需要重试，总成本可能更高

---

### 何时使用再生完整摘要

**最佳场景**：
- ✅ 摘要可解释性至关重要
- ✅ 会话有清晰的阶段边界
- ✅ 每次压缩时完整上下文审查是可接受的
- ✅ 需要生成人类可读的会话报告
- ✅ 压缩频率低（1-2 次）

**实际案例**：
- 需要生成会话总结给人类审查
- 阶段性任务（研究 → 规划 → 实现）
- 合规性要求可审计的摘要
- 教学或演示场景

**局限性**：
- 多次压缩周期中细节丢失
- 不适合需要精确文件追踪的场景
- 质量介于锚定迭代和不透明之间

## Claude Code 和 Deerflow 的压缩方法

### Claude Code 的方法

根据文档描述，Claude Code 使用的是**锚定迭代摘要的变体**：

**证据**：
1. **结构化摘要**：维护明确的章节（Session Intent、Files Modified、Decisions Made 等）
2. **增量合并**：后续压缩时合并新信息而非完全再生
3. **工件追踪**：专门的章节追踪文件操作
4. **三阶段工作流**：
   - 研究阶段：生成研究文档
   - 规划阶段：转化为实现规范
   - 实现阶段：基于规范执行

**特点**：
- 优化目标是 tokens-per-task，而非 tokens-per-request
- 使用示例工件作为种子（参考 PR、迁移示例）
- 在 70-80% 上下文利用率时触发压缩

**类型归类**：**锚定迭代摘要 + 三阶段压缩工作流**

## 实现建议

### 1. 选择压缩触发策略

| 策略 | 触发点 | 优势 | 劣势 |
|-----|--------|------|------|
| **固定阈值** | 70-80% 上下文利用率 | 简单、可预测 | 可能压缩过早 |
| **滑动窗口** | 保留最近 N 轮 + 摘要 | 上下文大小稳定 | 可能丢失早期重要信息 |
| **基于重要性** | 优先压缩低相关性部分 | 保留关键信号 | 实现复杂 |
| **任务边界** | 逻辑任务完成时压缩 | 摘要清晰 | 时机不可预测 |

**推荐**：滑动窗口 + 结构化摘要，在 70-80% 利用率时触发。

### 2. 设计摘要结构

根据你的智能体类型定制章节：

**编码智能体**：
```markdown
## Session Intent
## Root Cause (调试场景)
## Files Modified
## Decisions Made
## Test Status
## Current State
## Next Steps
```

**研究智能体**：
```markdown
## Research Question
## Key Findings
## Sources Consulted
## Hypotheses
## Next Research Directions
```

**通用智能体**：
```markdown
## User Goal
## Actions Taken
## Current Progress
## Pending Tasks
```

### 3. 实现增量合并

```python
def compress_context(existing_summary, new_messages, structure):
    # 只摘要新消息
    new_summary = summarize(new_messages, structure)

    # 逐章节合并
    merged = {}
    for section in structure:
        merged[section] = merge_section(
            existing_summary[section],
            new_summary[section]
        )

    return merged

def merge_section(existing, new):
    # 对于列表类章节（如 Files Modified），追加新项
    if is_list_section(existing):
        return deduplicate(existing + new)

    # 对于叙述类章节（如 Root Cause），保留两者
    else:
        return f"{existing}\n\nRecent update: {new}"
```

### 4. 单独追踪工件

由于工件追踪是所有方法的弱点，考虑单独维护：

```python
class ArtifactTracker:
    def __init__(self):
        self.files_read = set()
        self.files_modified = {}  # {path: [changes]}
        self.files_created = set()

    def on_file_read(self, path):
        self.files_read.add(path)

    def on_file_modified(self, path, change_description):
        if path not in self.files_modified:
            self.files_modified[path] = []
        self.files_modified[path].append(change_description)

    def get_summary(self):
        return {
            "read": list(self.files_read),
            "modified": self.files_modified,
            "created": list(self.files_created)
        }
```

### 5. 监控压缩质量

**关键指标**：
- **重新获取率**：压缩后智能体重新读取文件的频率
- **任务完成率**：压缩后任务成功完成的比例
- **Tokens-per-task**：完成任务的总 token 消耗

**警报阈值**：
- 如果重新获取率 > 20%，压缩可能过于激进
- 如果 tokens-per-task 增加 > 15%，压缩策略需要调整

## 常见问题

### Q1: 为什么不直接使用最高压缩率的方法？

**A**: 因为优化目标是 tokens-per-task，而非 tokens-per-request。

**示例**：
- 不透明压缩：节省 1000 tokens，但导致智能体重新读取 3 个文件（每个 5000 tokens）
- 总成本：-1000 + 15000 = +14000 tokens

- 锚定迭代：节省 930 tokens，智能体无需重新读取
- 总成本：-930 tokens

### Q2: 工件追踪为什么这么难？

**A**: 几个原因：

1. **累积效应**：长会话中文件操作可能有几十个，摘要难以全部保留
2. **细节层次**：需要记住"修改了什么"，而不仅仅是"修改了哪个文件"
3. **时间顺序**：同一文件可能被多次修改，需要追踪演变
4. **隐式操作**：有些文件被读取但未明确提及

**解决方案**：单独维护工件追踪器，不依赖摘要。

### Q3: 多少次压缩后质量会明显下降？

**A**: 取决于方法：

- **锚定迭代**：5-7 次压缩后仍保持良好质量（增量合并减少漂移）
- **再生完整**：3-4 次压缩后细节明显丢失（每次再生都有信息损失）
- **不透明**：2-3 次压缩后可能累积误差（依赖模型重建能力）

### Q4: 如何选择压缩触发阈值？

**A**: 推荐 70-80% 上下文利用率：

- **< 70%**：压缩过早，浪费上下文窗口
- **70-80%**：最佳平衡，留有缓冲
- **> 80%**：风险高，可能在关键时刻触发压缩

### Q5: 能否混合使用多种策略？

**A**: 可以！例如：

- **早期会话**：使用不透明压缩（信息少，压缩率重要）
- **中期会话**：切换到锚定迭代（信息累积，质量重要）
- **工件追踪**：始终单独维护

## 总结

### 核心要点

1. **优化目标**：Tokens-per-task，而非 tokens-per-request
2. **最佳方法**：锚定迭代摘要（质量最高，适合长会话）
3. **关键机制**：结构强制保留 + 增量合并
4. **最大弱点**：工件追踪（所有方法都需要改进）
5. **权衡**：额外 0.7% token 换取 10% 质量提升是值得的

### 快速决策树

```
需要最大 token 节省？
├─ 是 → 不透明压缩
└─ 否 → 会话 > 100 消息？
    ├─ 是 → 锚定迭代摘要
    └─ 否 → 需要人类可读摘要？
        ├─ 是 → 再生完整摘要
        └─ 否 → 锚定迭代摘要（默认选择）
```
