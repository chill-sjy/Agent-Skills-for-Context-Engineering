# 工件追踪问题与解决方案

## 概述

工件追踪（Artifact Trail）是上下文压缩中最具挑战性的维度。在所有压缩方法的评估中，工件追踪的得分普遍最低（2.19-2.45/5.0），远低于其他维度如准确性（3.35-3.70）和连续性（4.37-4.44）。这意味着即使是最先进的压缩方法，也难以可靠地保留编码智能体所需的文件操作历史。

本文档深入分析工件追踪为何成为最大痛点，为什么所有压缩方法在这个维度都表现不佳，以及如何通过专门的解决方案来改善这一问题。

---

## 一、为什么工件追踪是最大痛点

### 1.1 评分数据证据

根据对 36,611 条消息、数百个压缩点的评估，工件追踪维度的得分显著低于其他维度：

| 压缩方法 | 准确性 | 上下文感知 | **工件追踪** | 完整性 | 连续性 | 指令遵循 |
|---------|--------|-----------|------------|--------|--------|----------|
| 锚定迭代 | 3.70 | 4.04 | **2.45** | 4.44 | 3.80 | 4.99 |
| 再生完整 | 3.44 | 3.74 | **2.33** | 4.37 | 3.67 | 4.95 |
| 不透明压缩 | 3.35 | 3.43 | **2.19** | 4.37 | 3.77 | 4.92 |

**关键发现：**
- 工件追踪得分比其他维度低 **40-55%**
- 即使是表现最好的锚定迭代方法，工件追踪得分也仅为 2.45/5.0（不及格）
- 工件追踪是唯一一个所有方法都低于 3.0 分的维度
- 与连续性维度（4.37-4.44）相比，差距高达 **1.9-2.2 分**

### 1.2 实际案例：调试会话中的工件丢失

**场景：** 一个包含 178 条消息、89,000 tokens 的调试会话

**原始会话内容：**
- 用户报告 `/api/auth/login` 端点返回 401 错误
- 智能体读取了 `auth.controller.ts`（未修改）
- 智能体读取了 `middleware/cors.ts`（未修改）
- 智能体检查了 `middleware/auth.middleware.ts`（未修改）
- 智能体修改了 `config/redis.ts`（修复连接池配置）
- 智能体修改了 `services/session.service.ts`（添加重试逻辑）
- 智能体修改了 `tests/auth.test.ts`（更新 mock 设置）
- 运行测试：14 个通过，2 个失败

**压缩后的探针测试：**

问题："我们修改了哪些文件？请描述每个文件的更改。"

**良好响应（锚定迭代，得分 3/5）：**
> "我们修改了 config/redis.ts 来修复连接池配置，还修改了 session.service.ts 添加重试逻辑。测试文件也更新了。"

**问题：** 遗漏了 `tests/auth.test.ts` 的具体路径，没有提到读取但未修改的文件。

**较差响应（不透明压缩，得分 1/5）：**
> "我们修改了一些配置文件和服务文件来修复 Redis 连接问题。"

**问题：** 完全没有具体的文件路径，无法继续工作。

### 1.3 为什么工件追踪如此重要

对于编码智能体，工件追踪直接影响工作连续性：

**场景 1：继续未完成的工作**
- 用户："继续修复剩余的测试失败"
- 智能体需要知道：哪些测试文件已经修改过？哪些还没有？
- 如果工件追踪丢失：智能体会重新读取所有文件，浪费 tokens 和时间

**场景 2：代码审查和回顾**
- 用户："总结一下我们这次会话做了什么"
- 智能体需要知道：创建了哪些文件？修改了哪些文件？每个文件改了什么？
- 如果工件追踪丢失：智能体只能给出模糊的描述，无法生成有用的变更日志

**场景 3：调试回归问题**
- 用户："刚才的修改导致另一个测试失败了，回滚相关更改"
- 智能体需要知道：具体修改了哪些文件的哪些部分？
- 如果工件追踪丢失：智能体无法精确回滚，可能引入新问题

**成本影响：**
- 重新读取文件：每个文件 500-5000 tokens
- 重新探索架构：10,000-50,000 tokens
- 重新理解上下文：可能需要用户重新解释需求
- **总成本：** 工件追踪丢失可能导致 20-50% 的额外 token 消耗

## 二、为什么所有压缩方法在工件追踪上都表现不佳

### 2.1 根本原因分析

#### 2.1.1 信息密度不匹配

工件信息具有高密度、低冗余的特点：

```
对话内容类型对比：

【高冗余内容 - 易于压缩】
用户："这个错误是什么原因？"
智能体："让我检查一下日志..."
智能体："我发现问题了..."
智能体："问题是 Redis 连接..."
→ 可以压缩为："发现 Redis 连接问题"

【低冗余内容 - 难以压缩】
智能体修改了 src/config/redis.ts
智能体修改了 src/services/session.service.ts
智能体修改了 tests/auth.test.ts
→ 每个文件路径都是独立的事实，无法进一步压缩
→ 如果遗漏任何一个，信息就永久丢失
```

**问题：** 通用压缩方法会平等对待所有内容，导致高密度的工件信息被"稀释"在大量对话内容中。

#### 2.1.2 时间跨度问题

工件操作分散在整个会话中：
